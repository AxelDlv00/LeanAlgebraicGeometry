### Negative search results to record (do not retry)

* The symmetric-monoidal invertible-object API of `CommRing.Pic`
  (`Skeleton (SemimoduleCat R)ˣ` with the tensor-product monoid) does
  **not** lift to `X.Modules` because `X.Modules` carries no
  `MonoidalCategory` instance in Mathlib `b80f227`. There is no `tensorObj`
  for sheaves of `O_X`-modules at all, on either presheaves or sheaves.
  This kills the route literally suggested in `PROGRESS.md`
  ("`{ M : X.Modules // SheafOfModules.QuasicoherentData M ∧ ∃ N, Nonempty (M ⊗ N ≅ 𝟙_ _) }`")
  because the `⊗` symbol cannot be type-checked on `X.Modules`.
* `Skeleton X.Modules` exists (the skeleton of any category does), but
  without a monoidal structure it carries no `Monoid`/`CommGroup`, so the
  pattern used by `CommRing.Pic` (units of skeleton) cannot be reused.

### Adopted definition (first-approximation)

`LineBundle X := CommRing.Pic (X.presheaf.obj (op (⊤ : X.Opens)))`.

Justification recorded in the file's docstring. Briefly:

* **Non-vacuous.** Not `Unit` / `PUnit` / `X.Modules`.
* **Correct on affine schemes.** For `X = Spec R`,
  `LineBundle X = CommRing.Pic R = Pic(Spec R)` (Stacks 0AGS).
* **CommGroup is direct.** Inherited from `CommRing.Pic`'s commutative-group
  instance via `inferInstanceAs`. No new content needed.
* **Pullback is direct.** Composes `f.app ⊤` with the canonical iso
  `Γ(X, f ⁻¹ᵁ ⊤) ≅ Γ(X, ⊤)` (provable by `eqToHom` of `ext; simp`),
  yielding a ring hom `Γ(Y, ⊤) → Γ(X, ⊤)`, then applies
  `CommRing.Pic.mapRingHom`.
* **Affine pullback agreement.** For affine `X = Spec R`, `Y = Spec S`,
  `Pic.pullback f = CommRing.Pic.mapRingHom (Γf)` where `Γf : S → R` is the
  ring map corresponding to `f` — which is precisely the standard pullback
  `Pic S → Pic R`, `[N] ↦ [R ⊗_S N]`.

### Mathematical limitation (documented in file + here)

For non-affine `X`, this captures only the line bundles in the image of the
natural map `CommRing.Pic Γ(X, ⊤) → Pic(X)` — i.e. those that become
trivial after passing to the structure-sheaf component on global sections.
For projective space `P^n_k`, the global sections are `k`, so
`LineBundle P^n_k = Pic k = 0`, whereas the true Picard group is `ℤ`
(generated by `O(1)`). This will need to be fixed in Phase B/C step 2+.

The file's docstring spells this out in detail and records the two routes
the next iteration may take (build the tensor product on `X.Modules` and
upgrade, or replace by the relative-Picard-functor construction of
Phase C).
