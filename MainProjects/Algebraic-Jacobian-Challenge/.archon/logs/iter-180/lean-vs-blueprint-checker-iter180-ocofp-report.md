### `\lean{AlgebraicGeometry.Scheme.lineBundleAtClosedPoint.globalSections_iff}` (chapter: `lem:lineBundleAtClosedPoint_globalSections_iff`)
- **Lean target exists**: yes (L192–224)
- **Signature matches**: **NO** — see *Severity / must-fix-this-iter*. The Lean iff is:
  ```
  (order conditions on f)  ↔  Nonempty { s : H⁰(C, 𝒪_C(P)) // s ≠ 0 }
  ```
  The RHS quantifies existence of *some* nonzero global section and does **not** bind `s` to `f`. The blueprint prose pins the substantively different statement
  ```
  f ∈ K(C)^×  defines a global section of 𝒪_C(P)   ↔   (order conditions on f)
  ```
  i.e. the iff identifies `f` with a specific section via the inclusion `𝒪_C(P) ↪ 𝒦_C ≅ K(C)`. The Lean RHS is mathematically *vacuous in f*: since the canonical inclusion `𝒪_C ↪ 𝒪_C(P)` always provides a nonzero global section (the image of `1`), `Nonempty { s // s ≠ 0 }` is `True`. The statement therefore degenerates to
  ```
  (order conditions on f)  ↔  True
  ```
  for every nonzero `f`, which is false (take any `f` with a pole at some `Q ≠ P`: `ord_Q f = -1 < 0`, but the RHS is still true). The lemma as currently typed is FALSE as a mathematical claim.
- **Proof follows sketch**: N/A — body is `refine ⟨fun _h => ?_, fun _h => ?_⟩` then two `sorry`s. The comments correctly invoke Hartshorne II.7.7(a)/(b) and acknowledge each direction needs the dual-of-ideal-sheaf body of `lineBundleAtClosedPoint`; the *intent* matches the blueprint, but the typed statement does not.
- **Notes**: the prover's task_result flagged "mathematically odd RHS"; this is a real signature bug, not a stylistic concern. Fix shape options:
  - introduce a section-from-rational-function map (e.g. `lineBundleAtClosedPoint.ofFunction P hP f hf hord : H⁰(C, 𝒪_C(P))` once the body of `lineBundleAtClosedPoint` lands) and state the iff as `(order conditions on f) ↔ ∃ s, ι s = f` where `ι : H⁰(C, 𝒪_C(P)) →ₗ[k̄] K(C)` is the inclusion-into-function-field map; or
  - bind RHS via a `Σ`-shape: `Nonempty { s : H⁰ // ι s = f }`.
  The signature is NOT in `archon-protected.yaml` (only the three `Jacobian.*`, `Genus.genus`, and `AbelJacobi.*` declarations are protected), so the prover/plan agents are free to fix the type in-place.
